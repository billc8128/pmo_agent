# pmo_agent MVP — Design Spec

- **Status**: Draft, pending implementation
- **Date**: 2026-04-29
- **Author**: brainstormed with Claude (Opus 4.7) over a single
  session in the vibelive repo, then split out into this independent
  project

This spec is the **source of truth** for all design decisions made
during the brainstorming phase. When in doubt, this document wins;
update it when reality diverges.

---

## 1. Product summary

`pmo_agent` is a public timeline of a user's local AI-coding
sessions.

- A **daemon** runs on the user's machine, watches transcripts from
  local AI coding agents (Claude Code, Codex), and uploads each
  completed `(user_message, agent_response)` turn to a server.
- The server stores the turn, generates a one-sentence summary of
  the agent's response, and serves a public web profile.
- Anyone can visit `pmo_agent.example/u/<handle>` and see what that
  user has been doing with their AI coding agents, in near real
  time.

The MVP exists to validate **whether this artifact has value** —
both for the producer (do users want to publish this?) and for
viewers (do people want to read someone else's AI session log?).

### Out of scope for MVP

- Cursor support (its SQLite storage is significantly more complex
  to integrate than JSONL-based agents; deferred to v0.2)
- Privacy controls (private / unlisted timelines)
- Real-time push to viewers (web polls instead)
- Mobile / desktop client (web-only)
- Search, tagging, threading, comments, follows
- Backfill of historical conversations (only turns produced after
  the daemon starts running are captured)
- Tool result content (`tool_use` is summarized to a one-line
  description, `tool_result` is dropped entirely)

## 2. Architecture

```
┌─────── User's machine ───────┐         ┌────── Supabase ───────┐
│                               │         │                       │
│  pmo-agent daemon  (Go)       │         │  Postgres             │
│   ├─ watch ~/.claude/...      │ ─POST─► │   - users / turns     │
│   ├─ watch ~/.codex/...       │         │  Auth (PAT)           │
│   ├─ parse → redact (regex)   │         │  Edge Function:       │
│   └─ HTTPS upload (PAT auth)  │         │   - summarize via     │
│                               │         │     OpenRouter        │
└───────────────────────────────┘         └───────────────────────┘
                                                    │
                                                    ▼
                                          ┌─── Vercel ──────────┐
                                          │  Next.js (SSR)      │
                                          │   /u/:handle        │
                                          │   /me               │
                                          │   /discover         │
                                          └─────────────────────┘
```

Three layers, each replaceable:

- `daemon/` — Go single binary on the user's machine
- `backend/` — Supabase project (Postgres + auth + one edge fn)
- `web/` — Next.js SSR on Vercel

## 3. Data model

One table is enough for MVP.

```sql
-- backend/supabase/migrations/0001_initial.sql

create table public.profiles (
    id          uuid primary key references auth.users(id) on delete cascade,
    handle      text not null unique,                 -- public URL slug, e.g. /u/daisy
    display_name text,
    created_at  timestamptz not null default now()
);
comment on table public.profiles is 'Public-facing user identity. handle is the URL slug.';

create table public.turns (
    id                  bigserial primary key,
    user_id             uuid not null references auth.users(id) on delete cascade,
    agent               text not null,                  -- 'claude_code' | 'codex'
    agent_session_id    text not null,                  -- UI grouping key only, no state semantics
    project_path        text,
    turn_index          int  not null,                  -- 0-based position within the agent_session_id
    user_message        text not null,                  -- redacted plaintext
    agent_response_full text,                           -- redacted plaintext, may be large
    agent_summary       text,                           -- one-sentence LLM summary; null = pending
    user_message_at     timestamptz not null,
    agent_response_at   timestamptz,
    created_at          timestamptz not null default now(),
    updated_at          timestamptz not null default now()
);
comment on column public.turns.agent_session_id is 'Native session id from the agent (e.g. CC jsonl filename UUID). Used only for UI grouping; carries no active/closed state.';
comment on column public.turns.agent_summary is 'One-sentence LLM summary of agent_response_full, generated by an Edge Function.';

create unique index turns_dedup
    on public.turns (user_id, agent, agent_session_id, turn_index);

create index turns_user_time on public.turns (user_id, user_message_at desc);

-- Public read access to profiles and turns; writes only via PAT.
alter table public.profiles enable row level security;
alter table public.turns    enable row level security;

create policy "profiles are public"     on public.profiles for select using (true);
create policy "turns are public"        on public.turns    for select using (true);
create policy "users insert own turns"  on public.turns    for insert with check (auth.uid() = user_id);
create policy "users update own turns"  on public.turns    for update using (auth.uid() = user_id);
create policy "users delete own turns"  on public.turns    for delete using (auth.uid() = user_id);
```

Notes:

- One row = one **turn** (one user message + one agent response).
  This is the atomic unit. No "session" table; sessions are a
  derived view computed from `agent_session_id`.
- `agent_response_full` is **persisted** (not "use once and
  discard"). Storing the full response enables future "expand to
  see full response" UI, search, and knowledge sharing.
- Public-by-default via RLS: `select` is open to anonymous users.
- Writes are scoped to the authenticated user via RLS, so a stolen
  PAT can only pollute that user's own timeline.

## 4. Daemon (`daemon/`)

### 4.1 Responsibilities

- Watch `~/.claude/projects/**/*.jsonl` and
  `~/.codex/sessions/**/*.jsonl` for changes.
- Parse JSONL entries into completed `(user_message, agent_response)`
  turns. Detect turn boundaries (see 4.4 below).
- Apply local redaction rules to both `user_message` and
  `agent_response_full`.
- Upload completed turns to Supabase via REST.
- Maintain a local SQLite database (`~/.pmo-agent/state.db`)
  recording per-file byte offsets and per-turn upload status, so
  restarts don't double-upload.

### 4.2 CLI surface (v0)

Run in foreground only for v0; users start it in a terminal. No
`launchd` / `systemd` setup in MVP.

```
pmo-agent login           # paste PAT, store in ~/.pmo-agent/config.toml (mode 0600)
pmo-agent start           # start watching, foreground process
pmo-agent status          # show what's being watched and recent uploads
```

### 4.3 Storage

```
~/.pmo-agent/
├── config.toml           # PAT + server URL (mode 0600)
└── state.db              # SQLite: file_offsets, uploaded_turns, redaction_log
```

### 4.4 Turn boundary detection

Both Claude Code and Codex use append-only JSONL. The structure
varies per agent but the high-level entries are:

- a `user` entry containing one or more text/tool_result blocks
- one or more `assistant` entries, each containing some mix of
  `text` and `tool_use` blocks
- the cycle repeats: tool_result → assistant → ... until the
  assistant outputs a final block with no `tool_use`

A **completed turn** is detected when:

1. We've seen a fresh `user` entry containing a non-`tool_result`
   text block (this is the prompt).
2. After it, we've collected one or more `assistant` entries.
3. The most recent `assistant` entry contains no `tool_use` blocks.
4. At least 1.5 seconds have elapsed without new entries (debounce
   against streaming updates).

The collected `agent_response_full` is the concatenation of all
`assistant` text blocks in order, with `tool_use` blocks rendered
as one-line summaries:

```
[Tool: bash] ls -la ~/Desktop
[Tool: edit] foo.ts (lines 12-15)
```

`tool_result` content is **not** included.

### 4.5 Redaction (in daemon, before upload)

Redaction is the most important non-functional requirement of
this project. Failure is highly visible. Rules live in
`daemon/internal/redact/`.

Initial rule set (regex-based, all replace match with `[REDACTED]`):

- API key shapes: `sk_live_*`, `sk_test_*`, `xoxb-*`, `ghp_*`,
  `ghs_*`, `AKIA[0-9A-Z]{16}`, `eyJ[A-Za-z0-9_-]{20,}` (JWT)
- Email addresses
- IPv4 addresses (private ranges optional, public always)
- High-entropy base64-ish strings of length ≥ 32
- Lines starting with common env var patterns
  (`PASSWORD=`, `SECRET=`, `*_KEY=`, `*_TOKEN=`)
- Code-fenced blocks containing `.env` content (heuristic: block
  whose first line matches `KEY=value` pattern)

For every redaction, log `(rule_name, original_length, ts)` —
**not the matched content** — to the local SQLite for audit. This
lets users see "the daemon redacted N things today" without us
storing what was redacted.

### 4.6 Adapter abstraction

```go
package adapter

type Turn struct {
    Agent             string
    AgentSessionID    string
    ProjectPath       string
    TurnIndex         int
    UserMessage       string
    AgentResponseFull string
    UserMessageAt     time.Time
    AgentResponseAt   time.Time
}

type Adapter interface {
    Name() string
    Start(ctx context.Context, onTurn func(Turn)) error
    Stop() error
}
```

Implementations:

- `adapter/claudecode/` — fsnotify watcher on `~/.claude/projects/`
- `adapter/codex/` — fsnotify watcher on `~/.codex/sessions/`

Cursor adapter is **not implemented** in MVP.

## 5. Backend (Supabase)

### 5.1 Layout

```
backend/
└── supabase/
    ├── config.toml
    ├── migrations/
    │   └── 0001_initial.sql
    └── functions/
        └── summarize/
            └── index.ts
```

### 5.2 Auth

- Users sign up via Supabase Auth (email or OAuth — TBD; pick the
  simplest one supported by Supabase Auth out of the box, likely
  email-magic-link for v0)
- After signup, web app prompts for a `handle` and inserts a row
  into `public.profiles`
- "PAT" for the daemon = a long-lived Supabase Auth JWT issued via
  a service-role-signed token, OR (simpler) a custom row in a
  `public.tokens` table that the daemon presents in
  `Authorization: Bearer <pat>` header. We'll use the latter for
  v0; it's easier to revoke and audit.

### 5.3 The `summarize` Edge Function

Triggered by a Postgres webhook on `public.turns` insert.

```typescript
// backend/supabase/functions/summarize/index.ts (sketch)

import { serve } from "https://deno.land/std/http/server.ts";

serve(async (req) => {
  const { record } = await req.json();   // the inserted turn
  const summary = await summarizeViaOpenRouter(record.agent_response_full);
  // UPDATE turns SET agent_summary = $1 WHERE id = $2
  await supabaseAdmin
    .from("turns")
    .update({ agent_summary: summary })
    .eq("id", record.id);
  return new Response("ok");
});

async function summarizeViaOpenRouter(text: string): Promise<string> {
  const res = await fetch("https://openrouter.ai/api/v1/chat/completions", {
    method: "POST",
    headers: {
      "Authorization": `Bearer ${Deno.env.get("OPENROUTER_API_KEY")}`,
      "Content-Type": "application/json",
    },
    body: JSON.stringify({
      model: "anthropic/claude-haiku-4.5",   // cheap, fast, good enough
      messages: [
        { role: "system", content:
          "Summarize the assistant's response in one short sentence (under 20 words). " +
          "Focus on what was actually done or concluded. Output nothing else." },
        { role: "user", content: text },
      ],
      max_tokens: 60,
    }),
  });
  const json = await res.json();
  return json.choices[0].message.content.trim();
}
```

Notes:

- Model is `anthropic/claude-haiku-4.5` via OpenRouter for v0.
  Easy to swap by changing the string. OpenRouter is **the only**
  outbound LLM call site; the daemon never calls an LLM.
- If summarization fails (timeout, OpenRouter error, etc.), leave
  `agent_summary` as `NULL`. The web UI shows "Summary unavailable"
  with a "retry" button that re-invokes the function.

### 5.4 Why not call summarize synchronously from the daemon's POST?

We discussed this and chose **synchronous** during brainstorming
(daemon waits for summary before POST returns). Reconsidering for
the supabase architecture:

- Daemon → Supabase REST insert → Postgres webhook → Edge Function
  is naturally **asynchronous**. Forcing it synchronous would
  require the daemon to hit the Edge Function directly, bypassing
  the natural Supabase flow.
- Asynchronous works fine here because the user (and viewers) of
  the timeline don't care if the summary lags by a few seconds;
  the user message is already visible.

**Resolution**: go async. Daemon POST returns as soon as the
turn row is inserted. Summary is filled in within a few seconds
by the Edge Function. The web UI shows "Summarizing..." for turns
with `agent_summary IS NULL`.

This **revises** the synchronous-summary decision made during
brainstorming.

## 6. Web (`web/`)

Next.js (App Router), SSR via Vercel.

### 6.1 Routes

- `/` — landing page + install instructions
- `/signup` — Supabase Auth signup
- `/me` — own timeline + settings + PAT management
- `/u/:handle` — public timeline for any user
- `/discover` — community feed (public turns from all users,
  reverse-chronological, paginated)

### 6.2 Data fetching

- Server components fetch directly from Supabase using the public
  anon key (RLS enforces "public-read, authenticated-write" rules)
- Polling: `/u/:handle` and `/discover` poll the latest-turn
  endpoint every 10 seconds via a small client component. No
  websockets / realtime in MVP.

### 6.3 Turn rendering

```
14:08  [CC · vibelive]
  YOU      muted=true is TCC not authorized?
  AGENT    Yes — macOS denies camera access to Electron processes
           spawned from a non-Terminal context. Restart from
           Terminal to fix.
           [📄 expand]
```

- `YOU` block: full `user_message` text (already redacted).
- `AGENT` block: `agent_summary` by default. Click `[📄 expand]`
  to load `agent_response_full` and render it.
- Turns are grouped visually by `agent_session_id` (subtle
  separators), but each turn is independently rendered.
- "Summarizing..." placeholder when `agent_summary IS NULL`.

### 6.4 Discover page

A reverse-chronological list of turns from all users. Each item
shows:

- The user's handle (linked)
- The turn (same rendering as profile page)

Paginated, infinite-scroll. Anonymous viewers can read everything.

### 6.5 Account / settings

- Change handle / display name
- Generate / revoke daemon PATs
- (Future) toggle public / private; not in MVP

## 7. Implementation order

Recommended sequence; each milestone is independently demoable.

### Milestone 0 — backend skeleton (1-2 days)
- Set up Supabase project
- Apply `0001_initial.sql` migration
- Stub `summarize` edge function (returns "TODO" string, no
  OpenRouter call yet)
- Verify: insert a row via Supabase Studio, see it surface in a
  `select * from turns` query

### Milestone 1 — daemon end-to-end (3-5 days)
- `pmo-agent login` / `start` / `status` commands
- Claude Code adapter: parse JSONL, detect turn boundaries
- Local SQLite for offsets and upload state
- Basic redaction rules (API key shapes + email)
- POST to Supabase REST
- Verify: run daemon locally, have a real CC conversation, see
  rows appear in Supabase

### Milestone 2 — summarization (1-2 days)
- Wire `summarize` edge function to OpenRouter
- Trigger on insert via Postgres webhook
- Verify: turns get summaries within 5 seconds of upload

### Milestone 3 — web (3-5 days)
- Next.js app, deploy to Vercel
- `/u/:handle` route, server-side fetch from Supabase
- Polling for new turns
- "Expand" interaction for full response
- Verify: visit own profile in browser, see what the daemon
  uploaded

### Milestone 4 — Codex adapter (1 day)
- Mirror of Claude Code adapter; same JSONL parsing logic
- Verify: have a real Codex conversation, see it in profile

### Milestone 5 — discover + signup polish (2-3 days)
- `/discover` page
- Signup flow + handle picker
- `/me` PAT management
- Landing page

**Total**: ~2-3 weeks of focused work.

## 8. Open decisions to revisit

- **Auth method for end users**: email magic link vs OAuth. Pick
  whichever Supabase makes easiest by default.
- **PAT issuance flow**: how does a new user get their first PAT?
  Probably: after signup, web shows a one-time PAT and instructs
  the user to paste it into `pmo-agent login`.
- **Domain name**: TBD. Demo can run on
  `pmo-agent.vercel.app` initially.
- **Backfill historical conversations**: deliberately scoped out
  of MVP; revisit if early users ask for it.
- **Auto-start daemon (launchd / systemd)**: scoped out of MVP;
  ship a plist template and instructions for users who want it.

## 9. Decisions made during brainstorming (changelog)

The following decisions were made during the brainstorming session
on 2026-04-29. Each is the *current* answer; reasons in parens.

1. **Three coding agents to support**: started with Claude Code +
   Cursor + Codex; **scoped MVP down to CC + Codex** (Cursor's
   SQLite storage is significantly more work; revisit in v0.2).
2. **Atomic unit**: `(user_message, agent_response)` turn.
   Originally considered "session" granularity; revised after
   user clarified each turn should be visible.
3. **User message**: shown as raw text (redacted).
4. **Agent response**: shown as one-sentence LLM summary by
   default; full text expandable.
5. **Persistence**: agent response full text **is** persisted
   (not "use once and discard"). Reasoning: enables expand,
   search, knowledge reuse; demo doesn't have meaningful storage
   cost.
6. **Redaction**: regex-based, in the daemon, before upload.
   Audit log records rule + length, not matched content.
7. **LLM**: OpenRouter (model `anthropic/claude-haiku-4.5` for
   v0). Called only from the Edge Function.
8. **Summarization timing**: asynchronous (user message visible
   immediately; summary fills in within seconds). **Revised**
   from synchronous decision earlier in brainstorming, to fit
   Supabase architecture naturally.
9. **Privacy model**: public-by-default. Every profile is at
   `/u/:handle`, anonymous viewers can read everything. Privacy
   controls deferred.
10. **Real-time push to viewers**: not in MVP. Web polls every
    10s.
11. **No backfill**: only turns produced after daemon starts are
    captured.
12. **Tool handling**: `tool_use` rendered as one-line summary in
    `agent_response_full`; `tool_result` dropped.
13. **Daemon run model**: foreground only in v0. No `launchd` /
    `systemd` auto-start.
14. **Independent project**: `pmo_agent` is its own repo, its own
    git history, its own deployment. Not part of vibelive or any
    other project.
